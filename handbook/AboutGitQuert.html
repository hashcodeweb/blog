    常见的新手问题

    # 常见的新手问题

    判 sad 断仓库的远程 origin 使用的是 HTTPS 还是 SSH 地址

    git remote -v

    通过上面的命令可以查看远程仓库的origin 地址是否是 HTTPS 还是 SSH 格式

    以 git@github.com: 开头的地址表示是 SSH 格式的 Git 远程地址。如： origin git@github.com:username/repo.git

    如果远程地址是 HTTPS 格式,如: https://github.com/username/repo.git 则可以直接使用用户名和密码推送,不需要 SSH 密钥。

    # 不需要 SSH 密钥的情况一般有以下几种:

    1. 你是通过 HTTPS URL 来克隆仓库的,并且远程地址一直是 HTTPS 的。

    2. 你后来手动通过命令把远程地址改成了 HTTPS git remote set-url origin https://github.com/username/repo.git

    3. 你没有生成或者配置SSH密钥,所以Git自动使用HTTPS方式连接远程仓库。

    4. 如果远程地址是 HTTPS,但仍然需要SSH密钥的情况比较少见,可能的原因包括:

    配置了调试级别的Git日志,需要SSH密钥才能正常读取日志。 服务器配置问题,即使HTTPS也强制要求SSH密钥。 使用了一些工具或插件,它们要求SSH密钥才能正常工作。

    # 如何拉取分支项目

    git clone 命令克隆一个仓库的时候,它会拉取仓库的默认分支,也就是主分支 master。

    如果要clone一个仓库的其他分支,例如 gh-pages,可以在 clone 命令中使用 -b 参数指定分支:

    git clone -b gh-pages https://github.com/username/repo.git

    # git正确的工作流程

    1. 修改代码(比如编辑某个文件)
    2. 将修改过的文件使用 git add 添加到暂存区: git add 文件名
    3. 提交更新,创建新的提交: git commit -m "提交信息"
    4. 推送提交到远程仓库: git push

    # git使用用户名和密码的方式进行推送会报SSL错误

    fatal: unable to access 'https://github.com/hashcodeweb/blog.git/': OpenSSL SSL_read: Connection was reset, errno 10054

    git config --global http.sslVerify "false" 可以使用上面的命令来永久禁止ssl证书验证，但是不推荐 推荐使用ssh的密钥进行验证，可以避免SSL验证 也可以尝试更新git到最新版本

    # git默认推送方式

    1. 对于clone操作,Git默认使用HTTPS方式,需要输入用户名和密码。
    2. 对于push操作,Git默认使用SSH密钥方式。

    # 具体来说:

    • 当你从GitHub克隆仓库时,默认的URL是HTTPS格式的,如:https://github.com/user/repo.git。这需要使用GitHub用户名和密码来验证。

    • 但是,如果要往GitHub远程推送代码,默认配置下Git会使用SSH方式,如:git@github.com:user/repo.git。这需要在本地机器上配置SSH密钥,并在GitHub上添加部署公钥。

    • 如果你在clone时使用了SSH URL,那么在push时也会继续使用SSH。

    原因在于,clone只需要读取权限,使用用户名密码可以验证;但push需要写入权限,更安全的方式是使用SSH密钥来验证。

    可以通过git remote -v命令查看当前远程仓库的格式。要修改默认配置,可以用git remote set-url改变远程地址。

    所以综合来说,clone默认HTTPS,push默认SSH。

    # 如果我需要在本地更换账户，这个时候ssh需要更换吗

    是的,如果你需要在本地机器上切换不同的GitHub账号,那么对应的SSH keys也需要进行更换。

    # 主要的操作步骤包括:

    1. 为新的GitHub账号生成一对新的SSH Key。可以通过ssh-keygen命令生成。
    2. 在GitHub账户设置的SSH Keys页面里添加这个新生成的公钥。
    3. 在本地仓库配置中,修改远程origin的URL,将其改为新账号的SSH地址。可以通过git remote set-url修改。
    4. 删除本地机器上与旧账号相关的SSH key。通常在~/.ssh目录下。不再需要的key可以删除。
    5. 确保SSH客户端只加载与当前账户相关的私钥。可以通过ssh-add命令来指定私钥。

    这样就可以将本地repo与新账号通过SSH连接起来。之后使用git push等命令就会以新身份运行。

    需要注意的是,仅更换SSH key对于之前的commit历史没有影响,不会把旧账号的commit记录变成新账号。

    # repo是什么

    在Git中,“repo”是一个缩写,代表“repository”,中文意思是“仓库”。

    在版本控制系统Git中,仓库(repository)具体是指项目中的所有文件资源和Git操作记录的集合。

    每个Git仓库通常包含:

    • 项目中的所有文件和目录
    • 所有提交(commit)记录
    • 所有分支(branch)信息
    • 配置选项等元信息

    我们在使用Git时,经常会提到“本地仓库”和“远程仓库”:

    • 本地仓库(local repo):在本地机器上的Git仓库,包含项目代码和记录
    • 远程仓库(remote repo):存储在网络服务器或服务上面的仓库,例如GitHub

    在Git的操作中,我们可以通过将本地仓库和远程仓库关联,来实现代码的推送(push)和拉取(pull)。

    所以,“repo”可以理解为项目文件夹中的所有代码、历史和配置的集合。是进行版本控制的基本单位。

    知道这个词的意思,可以帮助理解和使用Git时的许多命令和操作。

    # 为什么git需要分三个区，modified staged commited

    Git需要分为modified、staged、committed三个区的主要原因有:

    可以分步提交修改,更灵活 Git允许你只将部分修改过的文件添加到暂存区,一个文件可以分多次提交到仓库。

    可以修改并回滚暂存的变更 在提交到本地仓库前,你可以随时查看和修改暂存的变更。

    维护工作目录的干净 工作区只包含当前未提交的修改,历史修改过的文件都在本地仓库中。

    用于代码审查与合作 可以将暂存的变更推送到远程分享给其他人,进行代码审查和协作。

    提交前预览修改 可以查看所有暂存的变更,对比不同版本,以确保提交是正确的。

    撤销本地修改 如果修改后反悔了,可以轻松撤销工作区的修改,不影响已提交的版本。

    保护已提交的历史版本 已提交到本地仓库的版本是不可变的,必须明确指出才能修改。

    所以这三个区系统可以让Git更安全可靠和灵活方便。

    # git add .之后会发生什么？

    在工作区对文件进行修改,这时文件内容存放在工作区中。 当执行 git add 时,文件内容会被生成一个对应的 blob 对象。 执行 git commit 时,会首先生成一个 tree 对象,tree Object 包含当前目录结构和每个文件对应的 blob 对象哈希值。 然后生成 commit 对象,commit 包含 tree 对象的哈希值、父提交指针、提交信息等数据。 此时文件内容在对应的 blob 对象中,树结构在 tree 对象中,提交信息存储在 commit 对象中。 blob 对象通常会进行压缩和重复利用,但不一定是在 commit 时进行。 commit 对象只是一个快照指针,不直接包含文件内容。 所以提交前后,文件内容都保存在 blob 中,commit 对象只保存元数据和指针

    # commit之后会发生什么

    会生成一个commit对象，该对象保存多个指针和信息，具体内容如下： 一个提交版本号(hash值-sha1生成的，主要用作产生随机数),用于唯一标识这个提交 作者信息(用户名,邮箱) 此次提交的说明消息 一个指向这个提交parent(上一个提交)的指针 一个指向项目源代码树(tight tree)的指针,代表此次提交项目的样子

    tree对象不包含文件内容,只保存文件名及对应blob对象的指针 blob对象包含文件内容,是一个可变对象，可以压缩文件的大小

    # 什么是多人协同作业

    通常的多人协作工作流程是:

    1. 所有参与者都从远程仓库(比如GitHub)clone代码到本地
    2. 每个人在自己的本地仓库中,从主分支切出一个新的分支
    3. 在该分支上进行开发,实现自己的功能
    4. 通过git add和git commit将修改提交到本地分支
    5. 当一个功能或任务完成时,可以推送(push)自己的分支到远程仓库
    6. 在GitHub上发起Pull Request,请求将你的分支合并到master主分支
    7. 团队成员 review代码,确认没有问题后,合并Pull Request
    8. 本地分支合并到master后可以删除,每个新任务再从master重新切分支
    9. 每个人定期在master分支上pull远程 master 分支,保持本地同步、

    这样通过分支发布、合并和同步,可以使多人高效协作而不会互相影响。

    # 修改同一个文件的事情是不是很少发生？

    关于这个问题,我的看法是:

    1. 这确实是一个比较少见的情况,但也并非完全不会发生。
    2. 在一个较大的项目或多人协作中,同时修改一个文件还是可能的。
    3. 一些核心文件如配置文件、主页等可能被多个人修改。
    4. 即使不常见,作为一个可能的情况,还是需要了解 Git 如何处理。
    5. 不管多么小心,人为错误导致的修改冲突还是会发生。
    6. 所以需要学习冲突解决以备不时之需。
    7. 只依赖“都很小心所以不会发生冲突”是不够的。
