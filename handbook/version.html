    版本控制

    冴羽 2023/11/11

    # GIT是什么？

    git 是分布式版本控制系统，工作的时候不需要联网。它是目前世界上最先进的分布式版本控制系统。

    # 什么是版本控制？

    是一种在开发的过程中用于管理我们对文件、目录或工程等内容的修改历史，方便查看更改历史记录，备份以便恢复以前的版本的软件工程技术

    • 实现跨区域多人协同开发
    • 追踪和记载一个或者多个文件的历史记录
    • 组织和保护你的源代码和文档
    • 统计工作量
    • 并行开发、提升开发效率
    • 追踪记录整个软件的开发过程
    • 减轻开发人员的负担，节省时间，同时降低认为错误

    无论是工作还是学习，或者自己做笔记，都经历过这样一个阶段，我们迫切需要一个版本控制工具。

    多人开发必须要使用版本控制，否则代价太大

    GIT svn cvs vss tfs 都是版本控制

    # 版本控制分类

    1. 本地版本控制

    2. 集中版本控制

    所有的数据都保存在服务器上，协同开发者从服务器上同步更新或上传自己的修改

    服务器一坏，那就坏了吧，刚好毕业了

    1. 分布式版本控制 所有版本信息仓库全部同步到本地的每个用户，这样就可以在本地查看所有版本历史，可以离线在本地提交，只需在联网时，push到相应的服务器或其他用户那里，由于每个用户那里保存的都是所有的版本数据，只要有一个用户的设备没有问题就可以恢复所有的数据，但这增加了本地存储空间的占用

    # git与svn最主要的区别

    svn时集中式版本控制系统，集中式版本控制系统是必须联网才能工作，对网络宽带要求比较高。

    # git的官网下载

    https://git-scm.com/

    打开下载好的安装包，无脑下一步

    # git的命令有哪些？

    都是linux的命令

    # git的配置

    用户名 邮箱

    # git的工作区域

    本地：工作目录（Working）、暂存区(Stage)、本地资源库(Repository) 远程：git仓库(Remote)

    Working-(git add files)-Stage-(git commit)-History-(git push)-Remote Remote-(git pull)-History-(git reset)-Stage-(git checkout)-Working

    # git 的六个基础命令与工作流程

    # 获取仓库的方法

    git init（初始化） clone（克隆远程仓库）

    # 提交：

    本地工作命令（add）到暂存区域 暂存区域（commit）到本地仓库 本地仓库（push）到远程仓库

    # 拉取项目

    远程仓库（clone）到本地仓库 本地仓库（checkout）到工作目录 远程仓库（pull）到工作目录

    # 免密码（ssh密钥）交互

    通过(ssh-keygen -t rsa)命令生成一段公钥 在github上输入自己的公钥，即可绑定成功

    # Git的分支

    默认 master主分支 查看分支:git branch 创建分支:git branch <分支名> 切换分支:git checkout <分支名> 创建+切换分支:git checkout -b <分支名> 合并某分支到当前分支:git merge <分支名> 删除分支:git branch -d <分支名> 修改分支名称:git branch -m <原名称> <新名称> 查看已合并和未合并的分支:git branch --merged | --no-merged 查看远程分支:git branch -r 查看所有分支:git branch -a 推送本地分支到远程:git push origin <分支名> 删除远程分支:git push origin --delete <分支名> 拉取远程分支到本地:git checkout -b <分支名> origin/<分支名>

    # 颜群

    # git是全量的

    就是说，每一个版本都包含全部的文件，时刻保持数据的完整性

    # git的文件状态

    工作区 暂存区 对象区

    # 设置用户名和邮箱

    --global （给当前用户配置） --system （给计算机配置） --local (给当前项目配置)

    # gitignore 配置需要忽略的文件

    通配符* dir/:忽略dir目录中的所有文件 dir/.txt dir//.txt 能够忽略 dir/abc/a.txt dir/xyz/a.txt dir/**/.txt 任意级别目录

    # 分支

    当前分支不能删除自己分支，可以在master分支删除分支

    # 什么是分支合并

    它指的是将一个分支上的修改合并到另一个分支的操作。例如： master分支有a文件 deve分支基于master,有a文件和新增的b文件 那么当在master分支上合并deve分支时: master分支现在会同时包含a文件和b文件 原因是在合并时,Git会把deve分支上所有有差异的提交都合并到master分支

    # 解决分支合并的冲突方式

    编辑文件a,直接查看两边的修改,决定最终想要的内容 删除特殊标记,完成文件编辑 添加修改后的a文件,然后提交 完成分支合并 合并有冲突时,我们需要手动决定保留每个文件中哪些修改,这就完成了分支的合并

    # 什么是fast forward？

    就是在分支合并（merge）时，master的commit会指向合并分支的最新commit 在merge时，默认使用的时fast，也可以禁止 （git merge --no-ff） 禁止fast可以保存分支合并的信息

    # 合并冲突怎么办？

    两个分支修改同一文件同一行 解决冲突还是git add commit -m （不是提交到版本库，而是告知冲突已经解决）

    # 分支合并的分类

    # 一种是没有冲突的合并

    默认情况下，git使用的是fast 当前分支合并到master分支，master分支的commit指针会指向当前分支最新的commit

    也有第二种（git merge --no-ff） 也就是禁止合并到同一个commit，这种方式会将master分支和当前分支合并到一起，并重新commit一次。

    # 一种是有冲突的合并

    有冲突的合并需要手动接触冲突

    # 版本穿梭

    commmit其实是一个对象链，也可以说是一个引用，每一次commit都对生成一个commit对象，master指针指向最新一次commit的commit的对象

    # 什么是版本回退？

    就是根据commit的对象链，将master主分支指向你想要回退到第几次的commit对象 命令是： 回退到n次： git reset --hard HEAD~n(n是数字) 回到指定sha1值： git reset --hard sha1的前几位

    # 回退到commit后，还可以后悔吗？

    可以通过git refog 查看记录，记录所有操作。

    # checkout 的放弃与游离操作

    可以查看以前commit的版本，但是不要修改，因为修改后必须提交

    # 版本穿梭

    1. 修改后，必须提交
    2. 创建分支的好时机

    如果修改了，影响不了后面的commit对象， 这个时候可以创建一个新分支

    # 分支重命名

    git branch -m master(原来的名字) master2(改成这个名字)

    # 保存现场

    (规范)

    1. 在功能没有开发完成前，不要commit

    2. 在没有commit前，不能切换分支 如果还没有将某一个功能开发完毕，就要切换分支： 建议：

    3. 保存现场（临时保存，stash） git stash

    4. 还原现场：git stash pop(将原来保存的删除，用于还原内容) git stash apply(还原内容，不删除原保存的内容) 说白了，就是删除掉原来保存的现场。。。有点像游戏的存档，你保存这个存档的时候，你每次玩都是从这个内容开始...

    5. 查看现场：git stash list
